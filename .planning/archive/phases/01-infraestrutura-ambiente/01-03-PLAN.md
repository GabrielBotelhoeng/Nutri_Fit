<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| WhatsApp → Evolution API | Mensagens de qualquer numero chegam sem autenticacao |
| Evolution API → N8N webhook | Qualquer sistema pode enviar POST para o endpoint /webhook/evolution |
| N8N → Backend Express | N8N encaminha payload da Evolution sem validacao de origem |
| N8N cron → Backend | Token interno necessario para autenticar chamada cron |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | Endpoint N8N /webhook/evolution | mitigate | Adicionar validacao do header "apikey" do Evolution API no workflow N8N (If node verifica se o header bate com EVOLUTION_API_KEY antes de processar). Implementar na Fase 2 quando o backend processar mensagens. |
| T-03-02 | Information Disclosure | Evolution API admin endpoint exposto | mitigate | AUTHENTICATION_API_KEY obrigatoria em todas as requisicoes. Em Railway (producao), a porta 8080 nao e exposta publicamente — apenas via URL Railway com HTTPS. Em dev local, exposta apenas em localhost. |
| T-03-03| Denial of Service | N8N webhook publico sem rate limit | accept | Para portfólio/MVP: volume de mensagens e baixo (1 nutricionista, poucos pacientes). Rate limiting adicionado quando volume justificar. |
| T-03-04 | Elevation of Privilege | N8N cron chamando backend sem autenticacao | mitigate | Cron workflow usa header "x-internal-token" com valor de variavel de ambiente N8N_INTERNAL_TOKEN. Backend valida esse token na rota /api/expiracao/verificar antes de executar qualquer logica. Implementado na Fase 4. |
| T-03-05 | Information Disclosure | evolution-api/.env com API keys | mitigate | evolution-api/.env no .gitignore raiz. Verificado via `git check-ignore -v evolution-api/.env` na Task 1. Railway usa variaveis de ambiente injetadas, nunca arquivo .env commitado. |
</threat_model>
